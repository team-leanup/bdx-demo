## 1. 제품 방향 전환

### 개요
BDX를 "상담 시스템"에서 **"고객 관리 CRM + 시술 기록 시스템"**으로 전환한다.

### 인터뷰 근거
- 고객 선호 데이터를 체계적으로 관리하지 못함 (기억/메모 의존)
- 시술 중 고객 특징을 계속 확인하는 피로도
- 과거 시술 디자인 사진 재검색 빈번
- 가격 오해로 인한 트러블 발생
- 기존 ERP(공비서 등)는 예약/매출 관리 중심 → 고객 성향 관리 부재

### 핵심 기능 3가지
1. **스케줄 관리 혁신** — 각 고객별 시술 성향 관리
2. **시술 사진 포트폴리오** — 과거 시술 디자인 빠른 검색
3. **고객 성향 CRM** — 태그 기반 고객 특성 데이터화

### 차별화 포인트
| 기존 ERP | BDX |
|----------|-----|
| 예약 관리 | 고객 성향 관리 |
| 매출 관리 | 시술 사진 데이터 |
| 텍스트 메모 | 태그/색상 직관적 표시 |
| 한국어만 | 다국어 상담 (ko/en/zh/ja) |

## 2. 홈 화면 / 스케줄 관리

### 개요
홈 화면을 스케줄표 중심으로 재구성. 각 예약 카드에 고객 이름, 시술 종류, 고객 특징 태그를 한눈에 표시한다.

### 상세 요구사항
- 스케줄표가 홈 화면에서 가장 크게 표시
- 예약 카드 구성:
  ```
  김지영
  자석젤
  [큐티클 민감] [오버레이 선호] [유지력 약함]
  ```
- 태그는 색상 코딩 (2.3 참조)
- 디자인 확정(🟢) / 현장 상담 필요(🟡) 아이콘 표시

### 기술 고려사항
- 기존 `(main)/home/` 페이지 리팩토링
- 고객 태그 데이터 모델 추가 (Zustand store 또는 data)
- 예약 카드 컴포넌트 신규 생성

### 히스토리
| 날짜 | 내용 |
|------|------|
| 2026-03-09 | 요구사항 정리 |
| 2026-03-10 | 홈 예약 카드가 `customerId` 기반으로 고객 주요 태그를 칩 형태로 표시하도록 반영. pinned 태그가 없을 때도 기존 고객 태그를 우선순위대로 노출 |

## 3. 포트폴리오 기능

### 개요
시술 사진을 그리드 형태로 관리하는 포트폴리오 페이지. 사진 클릭 시 시술 날짜, 고객 정보, 상담 기록, 시술 내용, 최종 가격을 확인할 수 있다.

### 사용 시나리오
1. 고객이 "예전에 다른 손님이 했던 네일 스타일" 요청
2. 포트폴리오에서 사진 검색
3. 당시 시술 구성과 가격 확인
4. 가격 오해 없이 시술 진행

### 상세 요구사항
- 사진 그리드 레이아웃 (Pinterest 스타일)
- 사진 상세 정보:
  - 시술 날짜
  - 고객 정보 (연결)
  - 상담 기록 (연결)
  - 시술 내용 (종류, 컬러, 파츠 등)
  - 최종 가격
- 검색/필터: 시술 종류, 기간, 고객, 가격대

### 기술 고려사항
- 기존 포트폴리오 사진 업로드 페이지 확장
- 이미지 저장: Vercel Blob 또는 외부 스토리지
- 포트폴리오 ↔ 고객카드 ↔ 상담기록 관계 설계

## 4. 고객 카드

### 개요
고객별 성향 정보를 태그로 관리. 강조, 색상, 상단 고정 기능 지원.

### 태그 예시
| 태그 | 설명 |
|------|------|
| 큐티클 민감 | 큐티클 관리 시 주의 필요 |
| 오버레이 선호 | 오버레이 적용 선호 |
| 두께 얇게 | 얇은 시술 선호 |
| 유지력 약함 | 자주 떨어짐, 보강 필요 |

### 고객 카드 기능
- 태그 추가/삭제
- 태그별 강조 표시 (✔)
- 태그별 색상 지정
- 중요 태그 상단 고정
- VIP 비주얼 카드: 마지막 시술 사진 미리보기
- 시술 이력 타임라인

### 기술 고려사항
- 기존 `(main)/customers/` 페이지 확장
- 고객 타입 정의에 태그 필드 추가
- 태그 시스템 공용 컴포넌트화 (스케줄 카드에서도 사용)

### 히스토리

|------|------|

## 6. 상담 UX 간소화

### 개요
기존 상담 플로우(고객정보 → 시술옵션 → 캔버스 → 저장)를 6단계로 재구성. 최대한 간단하게.

### 신규 상담 플로우
| Step | 내용 | 선택지 |
|------|------|--------|
| 1 | 시술 부위 | 손 / 발 |
| 2 | 시술 종류 | 원컬러 / 그라데이션 / 자석젤 / 아트 |
| 3 | 기본 조건 | 제거 여부 / 연장 여부 |
| 4 | 참고 이미지 | 업로드 |
| 5 | 고객 특이사항 | 큐티클 민감 / 오버레이 선호 / 유지력 약함 등 |
| 6 | 상담 저장 | 확인 + 저장 |

### 기존 → 신규 매핑
| 기존 | 신규 |
|------|------|
| consultation/customer/ | (예약에서 연결) |
| consultation/step1/ | Step 1-2: 부위 + 종류 |
| consultation/step2/ | Step 3: 기본 조건 |
| consultation/step3/ | Step 5: 특이사항 |
| consultation/canvas/ | 옵션 기능으로 분리 (9장 참조) |
| consultation/summary/ | Step 6: 저장 |

### 기술 고려사항
- 기존 `consultation/` 디렉토리 구조 변경
- Step 4 참고 이미지: 카메라/갤러리 업로드
- Step 5 특이사항: 고객 카드 태그 시스템과 연동
- i18n 키 전체 재구성

## 7. 가격 입력

### 개요
가격을 상담 단계가 아닌 **시술 종료 후** 마지막 단계에서 입력. 네일 시술 특성상 가격 변동이 많으므로 사후 확정 방식이 현실적.

### 가격 입력 구조
```
기본 가격: ₩ ____
+ 추가 옵션: ₩ ____  (파츠, 아트 등)
+ 추가 시술: ₩ ____
─────────────────
최종 가격: ₩ ____  → 확정
```

### 기술 고려사항
- 시술 완료 후 진입하는 별도 화면
- 상담 기록 + 고객 카드 + 포트폴리오에 가격 저장
- 가격 히스토리 (동일 고객 이전 가격 참조)

## 8. 상담 입력 방식

### 개요
상담 입력을 두 가지 경로로 제공한다.

### 방식 1: 예약 후 상담 링크 발송
- 네이버 예약 / 카카오 채널 예약 확정 후
- 고객에게 상담 링크 전달
- 고객이 모바일에서 시술 요청사항 미리 입력

### 방식 2: 매장 태블릿 모드
- 매장 대기 고객이 태블릿으로 상담 입력
- 기존 태블릿 대응 강화

### 기술 고려사항
- 링크 발송: 고유 URL 생성 (세션 기반)
- 태블릿 모드: 전용 레이아웃 / 키오스크 모드
- 두 경로 모두 동일한 상담 데이터 구조로 저장

## 10. 시술 리포트 기념품화

### 개요
시술 완료 후 **디지털 카드** 형태의 시술 리포트를 생성. 외국인 관광객에게 "여행의 추억"으로, 내국인에게는 "시술 기록"으로 활용.

### 리포트 구성
1. **비주얼 다이어리**: 매거진 레이아웃 (시술 전 → 디자인 시안 → 시술 후)
2. **K-뷰티 인증 마크**: "Certified by BDX" + 샵 낙관
3. **퍼스널 케어 팁**: 피부톤 맞춤 컬러 추천, 관리 팁 (다국어)
4. **QR코드**: 리포트 접근용

### 공유/마케팅 기능
- SNS 공유 버튼 (인스타 스토리용 이미지 생성)
- 비포/애프터 숏폼 영상 자동 생성 (15초)
- 친구 추천 할인 코드 (Referral)
- 샵 계정 + 해시태그 자동 삽입

### 기술 고려사항
- HTML → 이미지 변환 (html2canvas 또는 서버사이드)
- QR 코드 생성 라이브러리
- 이미지/영상 생성: 클라이언트 vs 서버
- SNS 공유 API 연동

## 11. 대시보드 개선

### 개요
매출 중심에서 **성과 중심 + 현장 중심**으로 대시보드 재구성.

### 신규 위젯/기능
| 위젯 | 설명 | 우선순위 |
|------|------|----------|
| 업셀링 리포트 | BDX 상담 후 추가된 옵션 매출 | P1 |
| 운영 효율 필터 | 확정/현장상담 아이콘 | P1 |
| 비주얼 퀵 메뉴 | 인기 메뉴 사진 썸네일 + 팝업 | P1 |
| 글로벌 대시보드 | 외국인 예약 수, 언어별 준비상태 | P1 |
| 재방문 골든타임 | 28일 주기 리마인더 대상 리스트 | P2 |
| 디자이너 지표 | 지정예약률, 상담완료율 차트 | P2 |

## 13. 인증 시스템 (구글 로그인)

### 개요
현재 BDX는 인증이 전혀 없는 프로토타입 상태. 실서비스를 위해 구글 OAuth 기반의 실제 인증 시스템을 구축한다.

### 현재 상태
- `src/app/(auth)/login/page.tsx` — Supabase Auth 이메일/비밀번호 로그인 후, 현재 샵의 온보딩 완료 여부에 따라 `/home` 또는 `/onboarding`으로 이동
- `src/app/(auth)/signup/page.tsx` — Supabase Auth 회원가입 후 빈 `shops` row와 owner `designers` row를 생성하고 즉시 로그인한 뒤 온보딩으로 연결
- `src/store/auth-store.ts` — Supabase `getSession()`과 `onAuthStateChange()`로 세션을 복원하고, `owner_id = auth.user.id` 기준으로 `currentShopId`와 원장 프로필을 복구함. 디자이너 PIN 비밀번호만 로컬에 보조 저장
- `src/lib/demo-account.ts` — 고정 데모 계정 credential을 정의하고, auth CTA에서 해당 계정으로 즉시 로그인 가능
- Supabase에는 `demo@bdx.kr` / `shop-demo` 데모 샵이 seeded 되어 있어 로그인 없이 체험하기로 바로 실제 예시 데이터 화면을 볼 수 있음
- 기존 `src/app/(auth)/lock/page.tsx` 잠금 화면은 제거되었고, 루트/가드/로그아웃 진입점은 모두 `/login`으로 통일됨
- 로그인 UX는 원장/직원 선택 대신 샵 계정 단일 진입으로 정리되었고, 디자이너는 로그인 후 앱 내부 프로필 개념으로 다뤄짐
- `src/middleware.ts` — 보안 헤더만 추가, 아직 Supabase 세션 검증은 하지 않음

### 기술 설계
1. **Supabase Auth 이메일/비밀번호 로그인**
   - 클라이언트에서 `signUp`, `signInWithPassword`, `getSession`, `onAuthStateChange` 사용
2. **샵 계정 ↔ 도메인 데이터 연결**
   - 회원가입 시 auth user id를 `shops.owner_id` 와 owner `designers.id` 에 연결
3. **샵 세션 복원**
   - 로그인/새로고침 시 `owner_id = auth.user.id` 로 샵을 조회하고 `currentShopId` 컨텍스트 복원
4. **추가 예정**
   - Google OAuth, 서버 검증 미들웨어, 비밀번호 재설정, 디자이너 개별 계정

## 14. 데이터베이스

### 개요
모든 데이터가 localStorage에 저장되는 프로토타입 → 실제 DB 도입. API 라우트를 통한 서버-DB 통신 구조로 전환.

### 현재 상태
- DB 관련 패키지 없음, `.env` 없음
- 데이터 흐름: `mock-*.ts` → Zustand → localStorage
- 이미지: base64로 localStorage에 직접 저장
- `shops`, `customers`, `consultation_records`, `booking_requests`, `portfolio_photos`는 이제 `shop_id` 기반 분리 구조를 사용하며, 신규 회원가입은 빈 샵으로 시작함

### 기술 설계
1. **DB**: Supabase PostgreSQL (Auth + DB + Storage 통합)
2. **ORM**: Prisma (`prisma/schema.prisma`)
3. **주요 테이블**: users, shops, customers, consultations, reservations, portfolio_photos, treatments
4. **API 라우트**: `app/api/` RESTful CRUD
5. **데이터 조회**: Zustand → SWR/React Query + API fetch
6. **이미지**: Supabase Storage (base64 → URL 참조)

### 마이그레이션 전략
1. DB 스키마 + API 라우트 먼저 구현
2. mock 데이터를 seed 스크립트로 DB 이관
3. Zustand store를 점진적으로 API 연동 전환
4. 전환 완료 후 mock 파일 제거

## 15. 개발 도구 / Supabase MCP

### 개요
개발 워크스페이스에서 Supabase MCP를 기본 조회 전용으로 연결해, 실수로 쓰기 작업을 실행할 위험을 낮춘다.

### 현재 상태
- 루트 `.mcp.json`에 Supabase MCP 원격 서버가 등록되어 있음
- 별도 `.env` 예시 파일은 없고, 첫 연결 시 클라이언트의 브라우저 인증 또는 수동 스코프 설정이 필요함

### 기술 설계
1. `.mcp.json`의 Supabase MCP URL에 `read_only=true`를 적용해 기본 모드를 조회 전용으로 제한
2. 최초 연결은 Supabase 공식 hosted MCP의 브라우저 인증 플로우를 사용
3. 특정 프로젝트로 범위를 제한해야 하면 `project_ref` 쿼리 파라미터를 추가하는 방식으로 확장

### 운영 메모
- 기본 URL: `https://mcp.supabase.com/mcp?read_only=true`
- 특정 프로젝트만 연결하려면 `https://mcp.supabase.com/mcp?project_ref=<PROJECT_REF>&read_only=true` 형태로 스코프 가능
- PAT 없이 브라우저 인증이 가능한 클라이언트에서는 추가 토큰 없이 로그인만 수행하면 됨
