## 1. 제품 방향 전환 — "상담 시스템 → 고객 CRM"

> 핵심: BDX는 ERP가 아니라 **고객 성향 관리 + 시술 기록 CRM**

### 1.1 ⬜ CLAUDE.md 제품 설명 업데이트
- **지금**: "네일 샵 상담 플로우 앱"으로 소개됨
- **할 일**: "네일 샵 고객 관리 CRM + 시술 기록 시스템"으로 변경. 핵심 기능 3가지(스케줄 CRM, 포트폴리오, 고객 성향 관리) 명시

## 2. 홈 화면 — 스케줄 중심으로 개편

> 핵심: 원장님이 앱 열면 **오늘 스케줄 + 고객 특징 태그**가 바로 보여야 함

### 2.1 🔵 홈 화면을 스케줄표 중심으로 변경
- **지금**: `GreetingHeader` + `TodayStatsCard` + `HeroCTA` + `QuickActions` 등 여러 카드가 나열됨. 예약 목록(`TodayReservationCard`)이 여러 카드 중 하나일 뿐
- **할 일**: 스케줄표(타임라인/리스트)를 화면의 **메인 영역**으로 올리고, 통계/CTA는 축소하거나 하단으로 이동

### 2.2 ⬜ 예약 카드에 고객 특징 태그 표시
- **지금**: 예약 카드에 고객명, 시술종류, 채널 뱃지만 표시. 고객 태그는 안 보임
- **할 일**: 각 예약 카드에 고객의 주요 태그를 칩(chip) 형태로 추가 표시
  ```
  김지영 · 자석젤
  [큐티클 민감] [오버레이 선호] [유지력 약함]
  ```
- **참고**: 고객 타입에 `tags: CustomerTag[]` 필드 이미 존재. 예약 데이터에서 `customerId`로 고객 태그 조회해서 렌더링

### 2.3 ⬜ 태그 색상 코딩
- **지금**: `CustomerTag` 타입에 `accentColor` 필드 있으나, 예약 카드에서 미사용
- **할 일**: 태그 카테고리별 기본 색상 자동 지정 + 사용자 커스텀 가능. 스케줄/고객카드/알림 등 전체에서 일관된 색상

### 2.4 ⬜ 상담 준비 상태 아이콘
- **지금**: 예약 카드에 상담 상태 표시 없음
- **할 일**: 각 예약에 🟢 디자인 확정 / 🟡 현장 상담 필요 아이콘 추가. 고객이 사전 상담 링크로 입력 완료했으면 🟢, 아니면 🟡

## 3. 포트폴리오 — 시술 사진 검색 시스템

> 핵심: "예전에 다른 손님이 했던 스타일" 요청 시 **사진으로 바로 찾고, 가격까지 확인**

### 3.1 🔵 포트폴리오 갤러리 강화
- **지금**: `(main)/portfolio/` 페이지 존재. 2~5열 그리드, 필터(전체/레퍼런스/시술), 고객명 검색 가능. 업로드 페이지도 있음
- **할 일**: 시술 종류/기간/가격대 필터 추가. 검색을 태그·컬러·디자인 타입으로 확장

### 3.2 ⬜ 사진 상세에서 시술 정보 + 가격 표시
- **지금**: `[id]/page.tsx` 상세 페이지 존재하나, 시술 상세·가격 연동이 부족
- **할 일**: 사진 클릭 시 아래 정보를 한 화면에 표시
  - 시술 날짜
  - 고객 정보 (이름, 태그)
  - 시술 내용 (종류, 컬러번호, 파츠)
  - **최종 가격** (기본 + 추가 옵션)
  - 연결된 상담 기록으로 이동 링크

### 3.3 ⬜ 포트폴리오 ↔ 고객 ↔ 상담 데이터 연결
- **지금**: `PortfolioPhoto` 타입에 `customerId`, `recordId?` 필드 있으나, 실제 연결 UI 미흡
- **할 일**: 포트폴리오 사진에서 고객 카드로 이동, 고객 카드에서 시술 사진 목록 확인, 상담 기록에서 시술 사진 확인 — 3방향 네비게이션

### 3.4 ⬜ 대시보드에 인기 시술 사진 썸네일
- **지금**: 대시보드 인기 시술 Top 3가 텍스트만 (하드코딩: "원컬러+포인트 42건")
- **할 일**: 실제 포트폴리오 사진 썸네일 배치. 클릭 시 컬러번호 + 가격 팝업

## 4. 고객 카드 — 성향 태그 관리

> 핵심: 고객 특성을 **기억이 아닌 데이터**로 관리. 강조·색상·고정 가능

### 4.1 🔵 고객 성향 태그 시스템
- **지금**: `CustomerTag` 타입 존재 (7개 카테고리: design/shape/length/expression/parts/color/etc). `pinned`, `sortOrder`, `accentColor` 지원. `tag-presets.ts`에 프리셋 정의
- **할 일**: 시술 현장에서 쓰이는 **주의사항 태그** 추가 (큐티클 민감, 오버레이 선호, 유지력 약함, 두께 얇게 등). 현재 태그가 시술 옵션 위주라, 고객 "체질/성향" 태그 카테고리 필요

### 4.2 ⬜ 태그 강조/고정 UI
- **지금**: 데이터 모델에 `pinned`, `accentColor` 있으나, 고객 카드 UI에서 직접 편집하는 기능이 부족
- **할 일**: 고객 상세 화면에서 태그별 ✔ 강조 토글, 색상 선택, 드래그로 순서 변경, 상단 고정 기능

### 4.3 ⬜ VIP 비주얼 카드
- **지금**: 고객 목록에 이니셜 아바타 + 이름 + VIP 뱃지 + 방문 횟수
- **할 일**: VIP 고객 카드에 **마지막 시술 사진** 작게 표시. 원장님이 고객 얼굴 대신 시술 사진으로 기억하므로, 한눈에 "아, 그 디자인 한 분" 인지 가능

### 4.4 ⬜ 시술 이력 타임라인
- **지금**: `treatmentHistory` 필드 존재 (날짜/부위/디자인/가격/이미지)
- **할 일**: 고객 상세에서 시술 이력을 **시각적 타임라인**으로 표시 (날짜별 사진 카드 나열)

## 5. 시술 전 알림 — 고객 주의사항 미리 확인

> 핵심: 시술 시작 전 "이 고객은 큐티클이 민감합니다" 같은 **팝업 알림**

### 5.1 ⬜ 시술 전 알림 모달
- **지금**: 예약 카드 → "상담 시작" 클릭 시 바로 상담 페이지로 이동. 고객 주의사항 확인 단계 없음
- **할 일**: "상담 시작" 클릭 시 → 고객 주요 태그(주의사항)를 모달/토스트로 먼저 표시 → 확인 후 상담 진입
  ```
  ⚠️ 김지영 고객 주의사항
  🔴 큐티클 민감 — 푸시백 시 주의
  🟠 유지력 약함 — 베이스 코트 2회 권장
  🟢 오버레이 선호
  ```

### 5.2 ⬜ 세이프티 태그 색상 코드
- **지금**: 태그 색상 시스템 없음 (2.3과 연계)
- **할 일**: 주의 수준별 색상 자동 지정 (🔴 높음 / 🟠 중간 / 🟡 참고 / 🟢 선호)

## 6. 상담 UX — 6단계로 간소화

> 핵심: 현재 상담이 복잡함. **최대한 빠르게 끝나야** 시술 가능 고객 수가 늘어남

### 6.1 ⬜ 상담 흐름 재구성 (현재 → 목표)

**현재 흐름 (5~7단계, 복잡):**
```
customer(고객정보) → step1(부위+오프+연장+쉐입)
→ step2(시술타입) → step3(파츠+색상수) → [canvas(선택)]
→ [traits(선택)] → summary
```

**목표 흐름 (6단계, 심플):**
```
Step 1: 시술 부위 (손/발)
Step 2: 시술 종류 (원컬러/그라데이션/자석젤/아트)
Step 3: 기본 조건 (제거 여부, 연장 여부)
Step 4: 참고 이미지 업로드
Step 5: 고객 특이사항 선택
Step 6: 상담 저장 (요약 확인)
```

- **할 일**: 기존 step1~3의 세부 옵션(쉐입 7종, 파츠 선택, 색상 수 등)을 줄이거나 옵션화. 고객정보 입력은 예약 연동으로 자동화. 캔버스는 옵션 분리(9장 참조)

### 6.2 ⬜ 기존 라우트 → 신규 라우트 마이그레이션
- **할 일**: `consultation/` 디렉토리 구조 변경, i18n 키 재구성, 상담 store 스텝 enum 수정

## 7. 가격 입력 — 시술 종료 후 확정

> 핵심: 네일 가격은 시술 중 변동이 많으므로, **시술 끝나고 최종 가격 입력**

### 7.1 ⬜ 시술 완료 후 가격 입력 화면
- **지금**: 상담 summary에 가격 관련 UI 없음. `price.ts` 타입은 존재 (기본가/추가옵션/총액)
- **할 일**: 시술 완료 후 진입하는 별도 가격 입력 화면 추가
  ```
  기본 가격:   ₩ 50,000
  + 추가 옵션: ₩ 10,000  (파츠 3개)
  + 추가 시술: ₩ 15,000  (연장)
  ─────────────────────
  최종 가격:   ₩ 75,000  [확정]
  ```

### 7.2 ⬜ 가격 히스토리
- **할 일**: 동일 고객의 이전 시술 가격을 참고로 표시. 포트폴리오/고객카드에도 가격 연동

## 8. 상담 입력 방식 — 링크 발송 + 태블릿

> 핵심: 상담 데이터를 모으는 **2가지 경로** 제공

### 8.1 ⬜ 고객 상담 링크 발송
- **지금**: 미구현
- **할 일**: 예약 확정 후 고유 URL 생성 → 카카오/문자로 발송 → 고객이 모바일에서 시술 요청사항 미리 입력. 입력 완료 시 예약 카드에 🟢 표시 (2.4 연계)

### 8.2 🔵 매장 태블릿 모드
- **지금**: 반응형 대응은 되어 있으나, 전용 "태블릿 키오스크 모드" 없음
- **할 일**: 대기 고객용 태블릿 전용 UI. 큰 버튼, 다국어 선택, 불필요한 네비게이션 숨김

## 9. 캔버스 → 옵션 기능으로 분리

> 핵심: 손가락별 디자인 캔버스는 **모든 샵이 필요한 건 아님**

### 9.1 ⬜ 캔버스 옵션화
- **지금**: `consultation/canvas/`에서 손가락별 색상·파츠 선택 가능. step3에서 "캔버스 열기" 버튼으로 선택적 진입은 가능
- **할 일**: 설정(settings)에서 캔버스 ON/OFF 토글 추가. OFF면 상담 플로우에서 캔버스 관련 UI 완전 숨김

## 10. 시술 리포트 기념품화 — 외국인 관광객 추억 콘텐츠

> 핵심: 시술 후 **"저장하고 자랑하고 싶은" 디지털 카드** 자동 생성

### 10.1 ⬜ 비주얼 다이어리 카드
- **할 일**: 매거진 레이아웃으로 [시술 전 손 상태 + 선택 디자인 시안 + 시술 후 사진] 배치. 핸드폰 사진첩에 저장 가능한 이미지 생성 (html2canvas 또는 서버)

### 10.2 ⬜ QR코드 리포트 접근
- **할 일**: 시술 완료 시 QR코드 생성 → 고객이 스캔하면 리포트 웹페이지 접근

### 10.3 ⬜ K-뷰티 인증 마크 / 샵 낙관
- **할 일**: "Certified by BDX" 로고 + 해당 샵 커스텀 낙관을 리포트에 삽입

### 10.4 ⬜ 퍼스널 케어 팁 (다국어)
- **할 일**: "당신의 피부톤에 어울리는 '누드 핑크' 컬러입니다", "7일 뒤 오일 발라주세요" 같은 개인화 케어 정보를 ko/en/zh/ja로 제공

### 10.5 ⬜ SNS 공유 (인스타 스토리용)
- **할 일**: 리포트 하단에 [인스타 스토리 공유] 버튼. 샵 계정 + #BDX #Kbeauty 태그 자동 삽입

### 10.6 ⬜ 비포/애프터 숏폼 영상
- **할 일**: 상담 사진 + 완료 사진으로 15초 Reels/TikTok용 영상 자동 생성

### 10.7 ⬜ 친구 추천 할인 코드 (Referral)
- **할 일**: 리포트 하단에 "이 리포트를 공유받은 친구 방문 시 5,000원 할인" 코드 삽입

## 11. 대시보드 — 매출 중심 → 성과·현장 중심

> 핵심: 원장님이 **"BDX 덕분에 돈 더 벌었네"**를 체감하는 지표

### 11.1 ⬜ 업셀링 리포트
- **지금**: 대시보드에 전체 매출만 표시
- **할 일**: "BDX 상담 후 추가된 옵션(파츠, 아트 등) 매출액"을 **상단에 별도 표시**. 원장님이 BDX 효과를 숫자로 체감

### 11.2 ⬜ 글로벌 대시보드
- **지금**: 다국어 관련 지표 없음
- **할 일**: [오늘의 외국인 예약: N명] + [언어별 상담 준비 완료] 상태 표시. BDX의 다국어 강점 시각화

### 11.3 ⬜ 재방문 골든타임 알림
- **지금**: 미구현
- **할 일**: "평균 주기 28일 기준, 이번 주 예약 알림 대상자 N명" 리스트. 클릭 시 리마인더 문구 복사

### 11.4 ⬜ 디자이너 지표 고도화
- **지금**: `DesignerPerformance` 위젯에 상담 건수 정도만 표시
- **할 일**: 매출 외에 **지정 예약률** + **상담 완료율** 바 차트 추가. 대형 샵에서 직원 관리 용도

## 13. 인증 — 구글 로그인 + 실제 인증 시스템

> 핵심: 현재 **인증이 전혀 없음** (로그인 버튼 누르면 검증 없이 통과). 실서비스를 위해 구글 OAuth 필수

### 13.1 ⬜ NextAuth.js (Auth.js) 설정
- **지금**: 로그인 페이지(`src/app/(auth)/login/page.tsx`)가 있으나, `handleLogin()`과 `handleGoogleLogin()` 모두 **검증 없이 `/onboarding`으로 바로 이동**. 실제 인증 라이브러리(NextAuth, Supabase Auth 등) 전혀 없음
- **할 일**: NextAuth.js (Auth.js v5) 설치 및 설정. `auth.ts` 설정 파일 + `app/api/auth/[...nextauth]/route.ts` API 라우트 생성

### 13.2 ⬜ 구글 OAuth Provider 연동
- **지금**: 구글 로그인 버튼 UI만 존재 (아이콘 + "Google로 시작하기")
- **할 일**: Google Cloud Console에서 OAuth 2.0 클라이언트 ID 발급 → NextAuth Google Provider 설정 → `.env.local`에 `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET` 추가

### 13.3 ⬜ 미들웨어 인증 가드
- **지금**: `src/middleware.ts`가 보안 헤더만 추가하고 **인증 여부를 전혀 검증하지 않음**. 로그인 안 해도 모든 페이지 접근 가능
- **할 일**: NextAuth 세션 기반으로 미들웨어에서 토큰 검증. 비로그인 시 `/login`으로 리다이렉트. 공개 라우트(login, splash, intro)만 예외 처리

### 13.4 ⬜ 인증 상태 관리 전환
- **지금**: `useAuthStore` (Zustand + localStorage)에서 `MOCK_DESIGNERS` 하드코딩 데이터로 모의 인증. 비밀번호도 단순 비트연산 해시(`'1234'`)
- **할 일**: Zustand 모의 인증 → NextAuth 세션 기반으로 전환. `useSession()` 훅 사용. 기존 `role`, `activeDesignerId` 등은 DB 유저 테이블에서 조회

## 14. 데이터베이스 — mock 데이터 → 실제 DB

> 핵심: 현재 **DB가 전혀 없고 모든 데이터가 localStorage**. 브라우저 초기화하면 데이터 전부 날아감

### 14.1 ⬜ DB 서비스 선정 및 연결
- **지금**: `package.json`에 DB 관련 패키지 없음 (Prisma, Drizzle, Supabase 등 전무). `.env` 파일도 없음. 모든 데이터가 `src/data/mock-*.ts` 하드코딩 → Zustand persist(localStorage)로 저장
- **할 일**: DB 서비스 선정 (Supabase PostgreSQL 추천 — 인증+DB+스토리지 통합) → 연결 설정 → `.env.local`에 DB URL 추가
- **대안**: PlanetScale(MySQL), Vercel Postgres, Neon 등

### 14.2 ⬜ DB 스키마 설계
- **지금**: 타입 정의만 존재 (`src/types/` — customer.ts, consultation.ts, portfolio.ts, auth.ts, shop.ts, price.ts 등)
- **할 일**: 기존 TypeScript 타입을 기반으로 DB 스키마 설계. ORM(Prisma 또는 Drizzle) 도입. 주요 테이블:
  - `users` — 디자이너/원장 계정 (구글 OAuth 연동)
  - `shops` — 매장 정보
  - `customers` — 고객 정보 + 태그
  - `consultations` — 상담 기록
  - `reservations` — 예약
  - `portfolio_photos` — 시술 사진
  - `treatments` — 시술 기록 + 가격

### 14.3 ⬜ API 라우트 구현
- **지금**: `src/app/api/` 라우트 없음. 모든 데이터 조작이 클라이언트 Zustand store에서 직접 처리
- **할 일**: Next.js App Router API 라우트(`app/api/`) 생성. CRUD 엔드포인트:
  - `/api/customers` — 고객 CRUD
  - `/api/consultations` — 상담 CRUD
  - `/api/reservations` — 예약 CRUD
  - `/api/portfolio` — 포트폴리오 CRUD
  - `/api/treatments` — 시술 기록 + 가격

### 14.4 ⬜ Zustand store → API 연동 전환
- **지금**: 6개 Zustand store가 각각 독립적으로 localStorage에 persist. 예:
  - `useCustomerStore` (`bdx-customers`) — mock 데이터 deepClone 후 localStorage
  - `useRecordsStore` (`bdx-records`) — 추가분만 persist, 조회 시 mock과 merge
  - `useReservationStore` (`bdx-reservations`) — 전체 persist
- **할 일**: store의 데이터 소스를 localStorage → API fetch로 전환. SWR 또는 React Query 도입 검토. localStorage는 오프라인 캐시 용도로만 유지

### 14.5 ⬜ 이미지 스토리지
- **지금**: 포트폴리오 사진, 참고 이미지가 **base64로 localStorage에 저장** (용량 한계, 성능 이슈)
- **할 일**: 외부 스토리지 서비스로 전환 (Supabase Storage, Vercel Blob, Cloudinary 등). base64 → URL 참조 방식으로 변경
