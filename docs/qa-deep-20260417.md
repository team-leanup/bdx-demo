# ❗ 가장 큰 논리 모순 3가지 (먼저 읽어주세요)

## 🔴 모순 1. "공유카드 보고 온 고객의 예약 날짜가 '오늘'로 강제 기록됨"

**문제 코드**: `src/lib/db.ts:1594~1613` (`dbCompletePreConsultation`)

**시나리오**:
1. 손님이 인스타에서 공유카드를 봄 → "미리 정하기" 버튼 클릭
2. 디자인·길이·컬러 선택 후 이름/전화 입력하고 제출
3. DB에 `booking_requests` 자동 생성되는데, **`reservation_date = 오늘 날짜`**, **`reservation_time = '미정'`** 으로 박힌다
4. 사장님 캘린더에 "오늘 미정 시간" 예약으로 나타남 → 실제 방문 원하는 날짜 아님

**왜 문제인가**:
고객은 "언제 방문할지"를 정한 적이 없는데, 시스템은 제출한 오늘을 방문일로 기록합니다. 사장님이 캘린더만 보면 "오늘 손님 오시는구나" 오해하고, 고객은 실제 원하는 날짜를 별도로 연락해야 합니다. 이중 소통 발생.

**권장 수정**:
- `reservation_date`를 NULL 허용으로 바꾸고 "날짜 미정" 상태를 UI에서 별도 표시
- 또는 완료 페이지에서 "원하시는 날짜" 입력란을 추가 받아 저장
- 또는 공유카드 플로우 자체에 간단한 날짜 선택 스텝 추가

## 🔴 모순 2. "사전상담 데이터가 두 테이블에 복사·저장되고 연결이 끊어진다"

**문제 코드**: `src/lib/db.ts:1556~1625`

**구조적 이상**:
- `pre_consultations.data`에 전체 상담 내용 저장

**왜 문제인가**:
1. 사장님 UI(`/records/preconsult/[bookingId]`)는 오직 `booking_requests.pre_consultation_data`만 읽음 → `pre_consultations` 테이블은 사실상 읽힌 적 없는 죽은 데이터

3. DB 용량·백업 비용 2배

**권장 수정**:
- 공유카드 경로에서 `pre_consultations` INSERT를 제거하고 `booking_requests`에만 저장
- 또는 반대로 `booking_requests.pre_consultation_data`를 지우고 `pre_consultations`만 사용, FK로 연결
- **지금 구조는 두 테이블의 책임이 겹침**

# 1️⃣ 공유카드 ↔ 사전상담 ↔ 예약 플로우

## 핵심 구조 이해

- **공유카드**: 별도 테이블 아님. `consultation_records.share_card_id` 컬럼 하나로 관리. 즉 **"완료된 시술 하나에 UUID 달아서 공유"**
- URL: `/share/{UUID}` → 시술 사진 + 디자인 정보 + 샵 정보만 보여줌 (예약 날짜·고객명 없음)
- CTA "미리 정하기" → `/pre-consult/{shopId}?from=share&designCategory=simple` (bookingId 없음)

## 두 가지 진입 경로

| | 경로 A (사장님이 예약 등록 후 링크 발송) | 경로 B (손님이 공유카드 보고 직접 접속) |

| bookingId | 있음 | 없음 |
| 예약 날짜 | 사장님이 등록한 날짜 | **오늘로 강제** (모순 1) |
| 디자이너 | booking.designer_id에 배정됨 | **null** |
| 고객 정보 | 자동 채움 | 고객이 직접 입력 |
| pre_consultations 저장 | X | O (이중 저장) |
| booking_requests 저장 | 이미 있음 (update만) | **신규 INSERT** |

# 2️⃣ 예약 → 상담 → 시술 완료 체인

## 발견된 문제 10건

### 🔴 CRITICAL
- **결함 1**: 링크 모달을 **열기만 해도** `consultation_link_sent_at`가 기록됨 (`ConsultationLinkModal.tsx:76~85`). 사장님이 실수로 모달 열었다 닫아도 "링크 발송됨" 상태가 되어, 고객이 링크를 못 받았는데 대기 상태가 됩니다.

### 🟠 HIGH
- **결함 2**: 고객이 사전상담 제출해도 `booking.customer_id`가 업데이트 안 됨 (`pre-consult/confirm/page.tsx:236~240`). walk-in 예약은 계속 customer_id=null인 채로 시술 기록됨 → 고객 카드에 이력 연결 안 됨
- **결함 3**: 같은 예약에 **사전상담 + 현장상담 둘 다** 진행하면 `consultation_records` 2개 생성. 하나는 화면에 안 보이는 **고아 레코드**가 됩니다 (`consultation/summary/page.tsx:198, 227`)
- **결함 8**: `visit_count` 이중 카운팅 가능. `records/[id]`에서 "결제 완료" 누른 뒤 `/payment`로 이동하면 `recordTreatmentCompletion`이 2번 호출될 수 있음

### 🟡 MED
- **결함 4**: 예약 취소해도 `pre_consultations` 테이블·연결된 `consultation_records`는 그대로 → 고아 데이터
- **결함 6**: 예약 디자이너 변경 후 상담 시작하면 `consultation_store`에 캐시된 이전 디자이너가 그대로 저장될 수 있음
- **결함 7**: `field-mode`에서 `finalizedAt`이 결제 시점이 아니라 **생성 시점**으로 세팅됨 → 실제 결제 완료 시각 추적 불가, `in_treatment` 단계 건너뜀
- **결함 9**: `wrap-up`에서 신규 고객 생성 시 FK 타이밍 이슈 가능 (고객 커밋 전에 record 업데이트 → FK 위반 위험)
- **결함 10**: "확정하기" 버튼이 `booking.status='confirmed'`로 바꿔도 `getBookingStage()`는 이 상태를 처리 안 함 → UI 변화 없고 무의미한 상태

## 발견된 문제 7건

### 🔴 CRITICAL
- **법적 리스크**: **약관 동의 내용이 DB에 저장되지 않음**. `sessionStorage`에만 저장되고, `dbCreateShopAccount`에도 `shops`/`designers` 테이블에도 동의 일시·내용 기록 없음 → GDPR/개인정보보호법 위반 가능 (`signup/page.tsx:67~71`)

- **온보딩 우회 가능**: URL 직접 접근으로 `/onboarding/complete`로 점프 가능. 포트폴리오 0장, 가격 미설정 상태로도 `onboarding_completed_at` 세팅 → 불완전 샵 오픈

# 5️⃣ Supabase DB 보안 (가장 시급)

## 🔴 CRITICAL — 즉시 수정 필요

### C-1. 익명 사용자가 **전체 샵 목록 조회 가능**
`shops_anon_select` 정책의 `qual = true` → 로그인 안 한 사람도 `SELECT * FROM shops` 가능. **전체 20개 샵의 owner_id, 전화번호, 주소, settings(민감 설정)** 노출.

### C-2. 익명 사용자가 **전체 사전상담 내용 조회 가능**
`pre_consultations_limited_select` 정책 `qual = true` → 모든 고객의 **이름, 전화번호, 상담 데이터 JSONB** 공개 상태. 개인정보보호법 위반.

### 즉시 수정 SQL
```sql
-- 전체 샵 공개 정책 제거
DROP POLICY IF EXISTS "shops_anon_select" ON public.shops;

-- 사전상담 공개 정책을 UUID 기반 직접 접근으로 제한
DROP POLICY IF EXISTS "pre_consultations_limited_select" ON public.pre_consultations;
-- UUID를 아는 사람만 접근 (RLS + where id = 방식) 또는 Edge Function으로 분리
```

## 🟠 HIGH

- **H-1/H-2/H-3**: `database.ts` 타입 정의에 `booking_requests.deposit`, `customer_tags.shop_id`, `pre_consultations.customer_id` 3개 컬럼 누락 → 코드에서 `as Record<string, unknown>` unsafe cast 사용 중 (`db.ts:710, 743, 786`)
- **H-4**: `auth.uid()`가 매 행마다 재평가되는 정책 10건 → `(select auth.uid())` 로 감싸면 성능 개선
- **H-5**: `shops`, `designers`, `customers` 등 핵심 테이블의 **초기 생성 마이그레이션 파일 부재** → `supabase db pull`로 베이스라인 생성 필요

## 🟡 MED

- **M-1**: 동일 테이블에 중복되는 permissive 정책 **169건** → OR 평가로 성능 저하
- **M-2**: FK 인덱스 **9개 누락** (`booking_requests.customer_id`, `consultation_records.shop_id` 등)
- **M-3**: `booking_requests.customer_id IS NULL`이 **29% (22건/75건)** → 고객 통계 집계 누락 위험
- **M-5**: Storage 버킷 3개(`designer-profile-images`, `portfolio-images`, `pre-consult-refs`) 전체 **파일 목록 listing 공개** → URL만 접근 허용으로 제한 권장
- **L-4**: `portfolio_photos_verified_insert`가 shop 존재만 체크하고 **본인 샵인지 검증 안 함** → 타인 샵에 포트폴리오 삽입 가능

# 💡 사용자에게 드리는 요약 (한 문단)

**지금 가장 큰 문제는 세 가지입니다.**
첫째, **Supabase 보안 정책 2개가 열려 있어 누구나 샵 목록과 전체 고객 사전상담 내용을 볼 수 있습니다** — 이건 당장 DROP POLICY로 막아야 합니다.
둘째, **공유카드로 들어온 손님이 제출하면 시스템이 예약 날짜를 '오늘'로 박아버려** 사장님 캘린더가 혼란스러워집니다. 원래 구조가 "예약 → 상담링크"인데 공유카드는 "예약 없이 들어와서 상담만 받는" 역방향이라 맞지 않습니다.
셋째, **회원가입 약관 동의가 DB에 저장되지 않아** 나중에 법적 문제가 생길 수 있습니다. 이 세 가지는 다음 주 안에 정리하시는 걸 권장드립니다.

나머지 60여건은 중·장기적으로 분기별 정리하시면 됩니다.
